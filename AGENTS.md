# AGENTS.md — Threads AI Bot

## Что это за проект

Это growth-движок для Threads веб- и диджитал-агентства Mononyx.

Система:

- публикует контент;
- принимает комментарии и упоминания под собственными публикациями;
- находит чужие публичные публикации по коммерческим фразам;
- классифицирует события как `lead`, `engagement` или `spam`;
- для подходящих лидов готовит мягкий CTA и передаёт контакт человеку через Telegram или WhatsApp.

Бот не закрывает сделки и не ведёт автономные продажи. Его задача — найти и квалифицировать интерес, затем передать тёплый контакт человеку.

Полное описание проекта и команды запуска находятся в `README.md`. Актуальная схема базы данных находится в `supabase/schema.sql`.

## Стек и ограничения

- Supabase (Postgres): база данных, RLS, RPC-функции и Cron.
- Supabase Edge Functions (TypeScript/Deno): короткоживущие фоновые задания.
- Cloudflare Workers (TypeScript): проверка и приём webhook от Meta.
- Python и GitHub Actions: только ручной резервный запуск заданий; постоянного сервера нет.
- Groq: LLM-классификация неоднозначных событий.
- Telegram Bot API: операторские уведомления.

Проект рассчитан на бюджет `$0` и использует только тарифы, которые официально допускают бесплатное коммерческое применение. Не добавлять платные сервисы, пробные платные тарифы или инфраструктуру, требующую банковскую карту, без явного решения человека. Перед добавлением нового внешнего сервиса проверять его актуальные условия использования.

Не менять выбранную архитектуру и не добавлять постоянный сервер без явного обсуждения.

- Не добавлять конфигурируемость, пока нет второго реального сценария её использования.

- Делать минимальное надёжное изменение и не расширять область задачи без необходимости.

## Архитектура

```text
Meta webhook
  -> Cloudflare Worker (verify signature, normalize, idempotent ingestion)
  -> Supabase: interactions / content_queue
  -> Supabase Cron -> Edge Functions:
       - Interaction Processor
       - Content Poster
       - Keyword Radar
  -> Threads API / Telegram после разрешённого выхода из shadow mode
```

Cloudflare Worker отвечает за ingestion. Классификация, повторные попытки и внешние действия выполняются короткоживущими Supabase Edge Functions. Python-задания и GitHub Actions сохраняются только как ручной резерв.

Основные директории:

- `worker/` — TypeScript Worker и его интеграционные тесты;
- `supabase/functions/` — Edge Functions и общие TypeScript-модули фоновых заданий;
- `supabase/migrations/` и `supabase/cron_*.sql` — инфраструктура и расписания Cron;
- `bot/` — резервные Python-клиенты и ручные фоновые задания;
- `supabase/schema.sql` — таблицы, индексы, RLS и атомарные RPC очередей;
- `supabase/verify_security.sql` — проверка RLS, прав таблиц и RPC;
- `.github/workflows/` — ручные резервные задания и ручной CI;
- `tests/` — Python-тесты;
- `config/keywords.json` — конфигурация Keyword Radar.

## Неприкосновенные инварианты

Не пересматривать эти правила без явного решения человека.

### Дедупликация и ingestion

- `source_item_id` — детерминированный ключ дедупликации: `reply:<id>`, `mention:<id>` или `keyword_search:<id>`. Не использовать случайные значения.
- Подпись Meta `X-Hub-Signature-256` проверяется по сырым байтам тела запроса до разбора JSON.
- Новые webhook-события вставляются через `INSERT ... ON CONFLICT DO NOTHING`.
- Повторная доставка события не должна изменять или откатывать статус уже существующей записи.
- Не доверять полям webhook payload до успешной проверки подписи и структурной валидации.

### Очереди и конкурентная обработка

- Получать задания только через RPC-функции `claim_interactions` и `claim_due_content`, использующие `FOR UPDATE SKIP LOCKED`.
- Не заменять атомарный claim на последовательность `SELECT` + `UPDATE` в клиентском коде: это создаёт гонку.
- Supabase Cron запускает Edge Functions по расписанию, но готовность повторной попытки и восстановление зависших заданий определяются данными в БД: `next_retry_at`, `processing_started_at` и таймаутом lease.
- Retry использует exponential backoff `2^attempts` минут, ограничение количества попыток и переход в `dead_letter`. Не использовать бесконечный retry или тихое удаление события.

### Публикация контента

- `content_queue` публикуется в два этапа: создание container, затем publish.
- Полученный `container_id` сохраняется в БД до publish, чтобы обеспечить восстановление и снизить риск повторного создания container.
- Не обещать строгую exactly-once публикацию, если её не гарантирует внешний API. Учитывать окно сбоя между ответом внешнего API и сохранением результата в БД.

### Внешние действия и shadow mode

- Любое новое поведение, отправляющее данные наружу, по умолчанию работает с `SHADOW_MODE=true`.
- К внешним действиям относятся ответы в Threads, публикации и сообщения в Telegram.
- Не выключать `SHADOW_MODE` самостоятельно.
- Включать реальные действия только по явному запросу человека после проверки на размеченных примерах.
- Keyword Radar никогда не отвечает под чужими публикациями автоматически. Он только сохраняет находку и отправляет операторский алерт, когда это разрешено текущим режимом.

## Безопасность и секреты

- RLS использует default-deny для `anon` и `authenticated` на всех таблицах и RPC.
- `service_role` — единственная прикладная роль с серверным доступом. Она используется только в Worker, Edge Functions и ручных резервных заданиях и никогда не передаётся frontend-коду.
- `SUPABASE_SERVICE_ROLE_KEY`, токены Meta, Groq, Telegram и другие секреты хранятся только в Cloudflare Secrets, Supabase Edge Function Secrets или GitHub Secrets для ручного резерва.
- Никогда не помещать секреты в код, комментарии, тестовые данные, логи, документацию, коммиты или сообщения чата.
- Не просить человека присылать значение секрета в чат. Просить настроить его самостоятельно через `wrangler secret put`, `supabase secrets set --env-file` или `gh secret set` для ручного резервного запуска.
- Перед изменением RLS, grants, RPC permissions или другой security-конфигурации выполнять `supabase/verify_security.sql`.
- Не ослаблять проверку подписи, RLS или права доступа ради упрощения локальной разработки.

## Работа с внешними API

- Не изобретать структуру Threads webhook payload по памяти.
- Сверяться с актуальной официальной документацией и, когда необходимо, с реально полученным тестовым событием.
- Если структура не подтверждена, явно обозначать предположение и не выдавать его за проверенный контракт.
- `LOG_WEBHOOK_PAYLOADS=true` включать только по явному разрешению человека и на минимально необходимое время.
- После проверки немедленно возвращать `LOG_WEBHOOK_PAYLOADS=false`.
- Не сохранять raw payload с персональными данными в репозитории, тестовых fixtures, отчётах или сообщениях.

## Проверка изменений

Перед завершением запускать проверки, относящиеся к изменённой области. При изменении общей логики или нескольких подсистем запускать полный набор. Изменения только в документации не требуют полного набора тестов.

```bash

python3 -m unittest discover -s tests -v
```

### Supabase Edge Functions

```bash
deno fmt --check supabase/functions
deno task --config supabase/functions/deno.json check
deno task --config supabase/functions/deno.json test
```

### Cloudflare Worker

Если зависимости ещё не установлены:

```bash
cd worker
npm ci
```

Проверки:

```bash
cd worker
npm run typecheck
npm test
npm run deploy:dry
```

### Supabase/Postgres

При изменении SQL:

- проверить применение `supabase/schema.sql` на чистом временном Postgres;
- после изменений RLS, grants или RPC permissions выполнить `supabase/verify_security.sql`;
- проверить атомарность `claim_interactions` и `claim_due_content`, если изменялась логика очередей.

### Критические сценарии

При изменении соответствующей логики отдельно проверять:

- дедупликацию по `source_item_id`;
- проверку webhook-подписи по raw body;
- атомарность `claim_*`;
- инкремент `attempts` и расчёт backoff;
- переход в `dead_letter` после исчерпания попыток;
- отсутствие внешних действий в shadow mode;
- восстановление публикации с сохранённым `container_id`.

Не считать задачу полностью проверенной на основании предположения «должно работать». Если проверку невозможно выполнить в текущем окружении, явно указать в итоговом отчёте причину, выполненные проверки и оставшийся риск.
